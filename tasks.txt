=== P3: Auth / Security / Beta Access ===

[x] 'group' creation locked behind code (for beta stage)
    DONE 2026-04-14 — Added BetaCode model to schema:
      BetaCode { id, code (unique), type ("registration"|"group_creation"),
                 usedById, usedAt, createdAt }
    Group creation route now checks GROUP_CREATION_BETA_REQUIRED env var.
    If "true", POST /groups requires betaCode in body; validates it is
    unused and of type "group_creation"; consumes it atomically.
    Admin code generation: POST /admin/beta-codes (requires X-Admin-Secret
    header matching BETA_ADMIN_SECRET env var). Returns array of codes.
    Public validation endpoint: POST /beta/validate (rate-limited 10/min)
    checks code validity without consuming it (for frontend pre-check UX).
    Frontend does NOT yet show a beta code input on the Create Group modal.

[ ] account registration locked behind code (for beta stage)
    PARTIAL — BetaCode model supports type:"registration" already.
    POST /beta/validate can check registration codes. But:
    - No registration UI exists yet (app uses dev-token only)
    - The actual registration endpoint (POST /auth/register) does not
      exist yet — needs to be built alongside the registration UI
    - When built, it should consume a registration BetaCode atomically

[ ] forgot password functionality
    PENDING — Requires:
    - POST /auth/forgot-password (generate time-limited token, send email)
    - Token storage (new PasswordResetToken model in schema)
    - POST /auth/reset-password (validate token, update password hash)
    - Frontend: ForgotPasswordPage, ResetPasswordPage routes
    - Blocked on: having password auth in the first place (currently
      using dev-token JWT which skips password entirely)

[ ] verify email functionality
    PENDING — Requires:
    - User.emailVerified Boolean field + User.emailVerificationToken
    - Send verification email on registration (or re-send endpoint)
    - GET /auth/verify-email?token=xxx route that sets emailVerified=true
    - Block certain actions (e.g. push subscription) for unverified users
    - Frontend: show "Please verify your email" banner for new users

[ ] verify account one other way (whatsapp, telegram, etc.)
    PENDING — Low priority until core email verify is done. Options:
    - WhatsApp Business API (paid, requires Meta approval)
    - Telegram bot (free, easier to integrate)
    - SMS via Twilio / Vonage
    Not started. Design decision needed before implementation.
