=== POST URL MIGRATION — Blocked until permanent subdomain is live ===
CONTEXT: The app currently runs behind a temporary ngrok URL. Many features
depend on WEB_BASE_URL and API_BASE_URL being set to permanent, publicly
accessible HTTPS URLs. The todo.txt has the infra steps (subdomain on
aidanlenahan.com, Cloudflare tunnel, etc.). Until those are done and the
.env values are updated to the real URLs, the following features are either
broken or incomplete for real users:

[ ] calendar feed URLs are ngrok-scoped and will break when tunnel restarts
    CONTEXT: GET /calendar/group-feed/:token.ics returns a URL built from
    API_BASE_URL. Currently this is http://localhost:4000 (or the ngrok URL).
    Once the subdomain is live:
    - Set API_BASE_URL=https://api.subdomain.aidanlenahan.com in apps/api/.env
    - Any already-generated calendarToken feed URLs in DB will still work —
      the token is permanent, only the host prefix changes. Users will need to
      re-copy the new feed URL (or you can regenerate all tokens).
    - webcal:// links in the GroupPage UI will also reflect the new host automatically.

[ ] ICS event links inside calendar files point to wrong host
    CONTEXT: buildIcsCalendar() uses WEB_BASE_URL to embed a "Open in Friendgroup"
    link in each VEVENT DESCRIPTION. Currently this is localhost:5173.
    Once subdomain is live: Set WEB_BASE_URL=https://subdomain.aidanlenahan.com

[ ] password reset emails contain localhost link
    CONTEXT: POST /auth/forgot-password builds resetUrl from WEB_BASE_URL.
    The link in the email will be http://localhost:5173/reset-password?token=...
    which is useless to anyone not on the dev machine. Fixed by updating WEB_BASE_URL.

[ ] join-request notification emails contain localhost group link
    CONTEXT: When a user joins a group, the owner receives an email with a link
    to /groups/:groupId?tab=members. This URL is built from WEB_BASE_URL.
    Same fix: update WEB_BASE_URL.

[ ] member approval notification emails contain localhost group link
    CONTEXT: Same as above — the "you've been approved" email links to
    WEB_BASE_URL/groups/:groupId. Useless until WEB_BASE_URL is real.

[ ] noreply email address is still Gmail SMTP — needs @subdomain address
    CONTEXT: Noted in todo.txt. Currently SMTP_USER is a Gmail address.
    After subdomain is live, set up noreply@subdomain.aidanlenaiah.com
    via Cloudflare Email Routing or a transactional mail provider.
    Update SMTP_USER, SMTP_PASS, EMAIL_FROM, SMTP_HOST, SMTP_PORT, SMTP_SECURE.

[ ] Google Calendar deep-link in EventPage embeds localhost in event URL
    CONTEXT: buildGoogleCalendarLink() includes webBaseUrl in the event details.
    Not a hard break (Google Calendar still works), but the "Open in Friendgroup"
    link inside the created GCal event goes to localhost. Fixed by WEB_BASE_URL.

[ ] group invite links (when built) must use permanent URL
    CONTEXT: todo.txt lists "make group invite link" as pending. When implemented,
    the shareable invite link must use the permanent subdomain, not ngrok/localhost.

=== P3: Feature Completeness / Secondary UX ===

[ ] persist user settings server-side for cross-device consistency
    CONTEXT: Currently, user settings that live in Zustand persisted stores
    (e.g. fg-duration-presets for event duration presets) are stored in
    localStorage only. They do not sync across devices — a user who configures
    their duration presets on mobile will not see them on desktop.
    WHAT TO DO:
    - Add a UserSettings model (or JSONB column on User) in Prisma schema to
      store arbitrary per-user preference blobs (e.g. { durationPresets: number[] })
    - Add API endpoints:
        GET  /users/me/settings  → returns settings blob
        PUT  /users/me/settings  → replaces/merges settings (validate shape with Zod)
    - On app load (after auth), fetch /users/me/settings and hydrate the
      relevant Zustand stores (overriding localStorage defaults)
    - On store mutation (addPreset / removePreset), write-through to the API
      so changes persist server-side immediately
    - Scope initially to durationPresets; design the schema generically so
      other future per-user prefs (e.g. notification defaults, default RSVP view)
      can be stored in the same blob without migrations
    EFFORT: small — JSONB on User is a 1-line migration; endpoints + store
    write-through are the bulk of the work.

[ ] dev vs prod instance — decide on deployment strategy
    DISCUSSION — Not a code task. Points to consider:
    - Current: single friendgroup_dev DB, local MinIO, local Redis
    - infra/docker-compose.yml + systemd services exist for deployment
    - Recommend: keep dev (friendgroup_dev), add prod (friendgroup_prod)
      with separate .env.production, MinIO bucket, Redis instance
    - Beta codes already gate access; decision needed before public push

[ ] recurring events
    PENDING — No recurrence model in schema. Needs RRULE support
    (rrule npm package), recurrenceRule field on Event, and UI for
    frequency/until/count in CreateEventPage. Not needed for initial beta.

[x] user registration flow — build POST /auth/register + RegisterPage UI
    DONE 2026-04-16 — Full registration flow implemented. See
    tmp/chat/4-16-user-registration-flow.txt for full implementation notes.
    Summary:
    - DB: added passwordHash, emailVerified, emailVerificationToken to User
      model (migration 20260416122825_add_auth_fields applied)
    - API endpoints added to apps/api/src/server.ts:
        POST /auth/register — firstName+lastName, email, password (8-32, rules),
          optional betaCode. Auto-generates username (firstName+lastName, lowercase,
          numeric suffix if taken). Creates unverified user, stores 6-digit OTP in
          Redis (10 min TTL), sends verification email (logged to console in dev).
          Returns { message, userId, emailSent }.
        POST /auth/verify-email — { userId, code } → verifies OTP, sets
          emailVerified=true, returns JWT token + user.
        POST /auth/resend-verification — { userId } → 60-second cooldown enforced
          via Redis TTL. Regenerates code and resends email.
        POST /auth/login — email + password, checks emailVerified (returns 403
          EMAIL_NOT_VERIFIED with userId if not), returns JWT on success.
        POST /auth/request-login-code — sends OTP to email (anti-enumeration:
          always returns 200). 60-second cooldown per user.
        POST /auth/verify-login-code — verifies OTP, also verifies email if
          not yet done, returns JWT token + user.
    - Frontend pages:
        RegisterPage.tsx — firstName (max 15), lastName (max 15), email (max 30),
          confirm email, password (8-32, uppercase+lowercase+number required),
          confirm password, optional beta code field. Auto-navigates to /verify-email
          on success.
        VerifyEmailPage.tsx — 6-digit numeric input, verify button, resend button
          (60-second cooldown countdown timer). Reads userId from query string.
        LoginPage.tsx — password form + "Other methods →" link that switches to
          email-code flow (request code → enter code → sign in). Both paths
          navigate to /groups on success.
    - App.tsx routes: /register, /verify-email added.
    - dev-token endpoint preserved for seeded dev users who have no password.
    PENDING items still open in this feature area:
    - Google SSO (noted as future work in spec)
    - Forgot password (separate P1 task)
